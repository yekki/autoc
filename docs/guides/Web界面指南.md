# AutoC Web 界面指南

> 版本 1.3 | 最后更新: 2026-02-25
> 关联设计文档: WEB_FRONTEND / ORCHESTRATOR / TOKEN_TRACKING

---

## 一、概述

AutoC Web 界面是一个基于 React + Ant Design 的可视化操作面板，提供与 CLI 完全对等的功能，并额外提供：

- 实时执行监控（SSE 事件流）
- 在线代码编辑器（Monaco Editor）
- 项目预览（iframe 内嵌）
- Docker 沙箱终端（xterm.js + WebSocket）
- Token 成本分析
- Bug 追踪与一键修复

### 访问方式

```bash
# 启动开发环境
./scripts/pm2-manage.sh dev

# 浏览器访问
http://localhost:3000
```

后端 API 运行在 `http://localhost:8080`，前端通过 Vite 代理自动转发请求。

## 三、项目仪表盘（首页）

打开 Web 界面或点击「返回所有项目」后进入首页。

### 功能

| 操作 | 说明 |
|------|------|
| **搜索项目** | 顶部搜索框，按项目名实时过滤 |
| **创建项目** | 点击「+ 创建项目」按钮，弹出创建对话框 |
| **导入 PRD** | 点击「导入 PRD」，支持拖拽上传或粘贴文本 |
| **进入项目** | 点击任意项目卡片进入工作台 |
| **编辑项目** | 卡片右上角菜单 → 编辑（修改名称、描述） |
| **删除项目** | 卡片右上角菜单 → 删除（含确认弹窗） |
| **批量删除** | 进入选择模式，勾选多个项目后批量删除 |

### 项目卡片信息

每张卡片展示：
- 项目名称和描述
- 状态标签（未开始 / 规划中 / 开发中 / 测试中 / 已完成 / 已暂停 / 未完成）
- 技术栈标签
- 任务完成进度
- 最后更新时间

### 创建项目对话框

| 字段 | 必填 | 说明 |
|------|------|------|
| 项目名称 | 是 | 显示名称 |
| 文件夹名 | 自动 | 默认与项目名相同，可自定义 |
| 项目描述 | 否 | 可留空，后续通过 AI 辅助润色 |

## 四、项目工作台

进入项目后，界面分为**侧边栏**和**主工作区**（6 个 Tab）。

### 4.1 侧边栏

#### 需求输入

侧边栏顶部是需求输入区域：

1. **输入框**: 输入自然语言需求描述
2. **运行模式**: 三种模式可选
   - **完整模式** (Full): PM → Dev → Test 全流程
   - **快捷模式** (Quick): 跳过 PM，直接开发
   - **修复模式** (Fix): 定向修 Bug
3. **高级参数**: 展开可设置
   - `max_rounds`: 最大迭代轮次
   - `max_iterations`: 最大迭代次数
   - `resume`: 断点续传
   - `clean`: 清理工作区
   - `no_parallel`: 禁用并行
   - `verbose`: 详细日志
4. **运行按钮**: 点击开始执行

#### 技术栈编辑与推荐

点击「编辑」进入编辑态后，需求输入框下方出现技术栈选择区域：

- **手动选择**：从下拉列表中选择已知技术栈（后端/前端/数据库/工具），或直接输入自定义标签回车添加
- **智能推荐**（推荐）：点击「⚡ 智能推荐」按钮，系统即时根据需求描述中的关键词本地匹配合适的技术栈，**无网络请求，零延迟**
- **AI 精准**（可选）：点击「AI 精准」按钮，调用「辅助」Agent 配置的模型对需求进行深度分析并推荐技术栈，耗时 1~5 秒，消耗少量 Token。若未配置辅助模型，此按钮不显示

> 两种推荐方式均为增量追加，不会覆盖已有选择。技术栈信息会随项目保存，并在执行时传递给 PlanningAgent 作为约束。

#### 任务列表

执行过程中，侧边栏中间会显示 PlanningAgent 规划的任务列表：
- 每个任务显示状态图标（待定 / 进行中 / 已完成 / 已验证）
- 点击任务可查看详情

#### Token 面板

侧边栏底部显示实时 Token 消耗和任务进度条。

### 4.2 概览 Tab

根据项目状态自适应显示四种视图：

| 状态 | 显示内容 |
|------|---------|
| **初始化** | 沙箱准备进度（Docker 容器启动），SessionHeader 右上角显示「停止」按钮 |
| **执行中** | 圆形进度图 + **当前任务名和阶段**（如"task-2 Markdown 渲染与标签系统 · 代码开发与实现 · 已完成 1/3"）、Agent 实时思考内容（支持折行展示），SessionHeader 右上角显示「停止」按钮 |
| **已完成** | 执行摘要（任务/测试/Token/耗时），SessionHeader 右上角显示「追加功能」等操作按钮 |
| **空闲** | 项目概况、历史会话，SessionHeader 右上角显示「继续执行」「快速修复」等操作按钮 |
| **空闲 — 开始开发** | 项目无历史会话时显示快捷开发表单，包含需求输入框（支持 AI 润色）和技术栈选择（支持智能推荐 / AI 精准推荐） |

> **AI 辅助功能**（概览 Tab 空闲态 + 侧边栏编辑态）：
> - **AI 润色**：对简短的需求描述进行扩展优化，生成包含核心功能、目标用户、关键交互的完整需求文案（100-300 字）。**需要在设置中为「辅助」Agent 分配模型，未配置时按钮不显示。**
> - **智能推荐**：基于需求关键词本地匹配技术栈标签（零延迟、无 Token 消耗，始终可用）
> - **AI 精准推荐**：调用「辅助」Agent 配置的模型分析项目类型后推荐最合适的技术栈组合（消耗少量 Token）。**同样依赖辅助模型配置，未配置时按钮不显示。**
>
> **Markdown 渲染**：概览 Tab 顶部的主需求卡片支持 Markdown 格式渲染（标题、列表、表格、代码块等），输入的需求描述中如果包含 Markdown 语法会自动格式化展示。点击标题栏可折叠/展开主需求内容，折叠时显示首行摘要。
>
> **状态保持**：所有折叠/展开操作（主需求、版本时间线、执行记录、会话详情等）会自动保存到 localStorage，刷新页面后恢复到用户最后操作的状态。

### 4.3 代码 Tab

内置 Monaco 编辑器（VS Code 同款），功能：

| 功能 | 说明 |
|------|------|
| **文件树** | 左侧显示项目所有文件，点击切换 |
| **语法高亮** | 自动识别语言 |
| **在线编辑** | 可直接修改代码 |
| **保存** | `Ctrl+S` 保存修改到项目 |
| **Diff 模式** | 对比修改前后的差异 |
| **全屏** | 编辑器全屏模式 |
| **NEW 标记** | 执行过程中新创建的文件会标记 NEW |

### 4.4 预览 Tab

流水线执行完成后，如果生成的是 Web 应用，系统自动启动预览服务：
- 通过 iframe 内嵌显示应用
- 支持手动刷新
- 显示预览 URL（可在新窗口打开）

### 4.5 终端 Tab

基于 xterm.js + WebSocket 的终端，连接到项目的 Docker 沙箱容器：
- 可直接在沙箱内执行命令
- 查看运行日志
- 调试代码

### 4.6 需求历史 Tab

按**需求 → 会话**两级结构展示项目的所有执行历史。

#### 一级：需求卡片

每个不同的需求描述独立成卡：

| 元素 | 说明 |
|------|------|
| **标签** | `当前需求`（蓝色边框高亮）或 `历史需求` |
| **时间范围** | 精确到秒，格式 `02/25 02:52:30 ~ 02/25 06:01:15` |
| **统计** | 执行次数 / 成功数 / 失败数 / 累计 Token |
| **需求正文** | 完整需求文本，支持 Markdown 格式渲染，长文本支持展开/收起（折叠状态跨刷新保持） |
| **技术栈** | Python / Flask / SQLite 等标签 |

> **智能归组**：`恢复执行`、`重新测试`、`Bug 修复` 等操作会自动归入它所属的原始需求卡片下，不会独立成卡。

#### 二级：执行记录

点击「执行记录 (N)」展开该需求下的所有会话（展开/折叠状态自动保存，刷新页面后保持）：

| 元素 | 说明 |
|------|------|
| **操作类型** | 彩色标签：完整执行 / 恢复执行 / Bug 修复 / 需求调整 / 定向修复 / 新增功能 / 重新测试 |
| **状态** | 成功 ✓ / 失败 ✗ 图标 |
| **任务进度** | `2/3 任务`，成功时绿色高亮 |
| **耗时** | 精确到秒或分秒格式 |
| **Token** | 本次消耗量，无消耗时灰色弱化显示 |
| **失败原因** | 红色背景内联展示（如有） |
| **Agent 分布** | PM / 开发 / 测试 各自 Token 消耗 |

#### 会话管理

- **删除单条**：每行右侧删除按钮（含确认弹窗）
- **清理全部**：右上角「清理全部」按钮，清除所有已结束的会话

### 4.7 成本 Tab

| 内容 | 说明 |
|------|------|
| **Agent 分布** | PM / Developer / Tester 各自 Token 消耗占比（优先 `executionAgentTokens`，兜底从迭代历史按 phase 汇总） |
| **项目累计** | 总消耗、执行次数、平均每次、输出/输入比 |
| **Token 分布** | 新输入 / 缓存命中 / 输出 三段条形图 + Prompt Caching 节省卡片 |
| **执行历史** | 每次执行的 Token 消耗条形图（有明细时显示 I/O/缓存三段条） |
| **模型估算** | 基于 GLM 三维定价（input/cache_read/output），每个 Agent 按自身 token 数 × 对应模型单价独立估算 |
| **AI 辅助** | 描述润色 / 技术栈推荐的独立 Token 消耗（紫色卡片，标注"非执行"） |

**侧边栏 Token 面板**（`TokenPanel`）：显示项目 Token 总量 + 最近一次增量 + 基于 GLM 定价的费用估算，点击跳转到成本 Tab。

## 七、项目操作速查

| 操作 | Web 界面入口 | 等价 CLI |
|------|-------------|---------|
| 创建项目 | 首页「+ 创建项目」 | `autoc init my-project` |
| 导入 PRD | 首页「导入 PRD」 | `autoc import-prd file.md` |
| 运行开发 | 侧边栏输入需求 → 运行 | `autoc run "需求"` |
| 快捷开发 | 切换到 Quick 模式 | `autoc quick "需求"` |
| 修复 Bug | 切换到 Fix 模式 | `autoc fix "描述"` |
| 停止执行 | 概览 Tab SessionHeader「停止」按钮 | `Ctrl + C` |
| 重新测试 | 历史 Tab → 重测按钮 | `autoc retest` |
| 快速修复 | Bug 列表 → 修复按钮 | `autoc fix "bug描述"` |
| 需求调整 | 需求调整按钮 | — |
| 编辑代码 | 代码 Tab → 编辑 → Ctrl+S | 手动编辑文件 |
| 删除项目 | 卡片菜单 → 删除 | `autoc delete name` |
| 配置模型 | 设置抽屉 | `autoc model set ...` |

### 操作按钮对需求与计划的影响

五种核心操作对项目需求和实现计划有不同的影响，理解这些区别有助于选择正确的操作：

| 操作 | 对主需求 | 对实现计划 (PLAN.md) | 对已有代码 | 版本变化 | 适用场景 |
|------|---------|---------------------|-----------|---------|---------|
| **继续执行** (Resume) | 不变 | 不变（从文件恢复） | 保留 | 不变 | 执行中断后恢复、已完成项目重跑验证 |
| **快速修复** (Quick Fix) | 不变 | 不变 | 局部修改 | +0.0.1 | 修复具体 Bug，不动架构 |
| **追加功能** (Add Feature) | **不变**（保持主需求不动） | **归档旧 plan → 生成新 plan** | 保留 | +0.1.0 | 在主需求基础上增量开发新功能 |
| **重新定义** (Redefine) | **替换为新需求** | **删除 → 全部重新生成** | **全部清空** | +1.0.0 | 项目目标彻底改变，从头开始 |
| **需求调整** (Revise) | 按 `cleanWorkspace` 委托 | 同上两者之一 | 同上 | 同上 | 兼容旧 API 入口 |

**追加功能的上下文保护**：点击「追加功能」时，系统会自动保护主需求的上下文。PM 在规划新功能时能看到主需求原文和已有架构，Coder 在实现时也知道"项目的核心目标是什么"，从而避免新增功能破坏已有功能。旧 plan 会自动归档到 `plan_history`，可在工作区找到 `PLAN-v{版本号}.md` 查看历史版本。

> 技术详情见 [编排器.md](../design/编排器.md) §3.1 次需求上下文保护机制。

## 八、SSE 实时事件

Web 界面通过 SSE (Server-Sent Events) 接收执行进度，关键事件类型：

| 事件类型 | 含义 | UI 响应 |
|---------|------|--------|
| `phase_start` | 阶段开始（PM/Dev/Test） | 更新阶段指示器 |
| `plan_ready` | PM 规划完成 | 显示任务列表 |
| `task_start` / `task_complete` | 任务开始/完成 | 更新任务状态 |
| `task_verified` | 任务验证结果 | 通过 → `verified`；失败 → `failed` + 展示具体原因 |
| `test_result` | 测试结果 | 更新 Bug 列表 |
| `file_created` | 文件创建 | 追加到文件树 |
| `token_session` | Token 统计 | 更新成本面板 |
| `sandbox_prepare` / `sandbox_ready` | 沙箱状态 | 更新概览进度 |
| `preview_ready` | 预览就绪 | 显示预览面板 |
| `done` | 执行完成 | 关闭 SSE，清理残留 `in_progress` 任务，刷新数据 |
| `error` | 执行出错 | 显示错误信息和恢复建议 |

SSE 支持断线自动重连（最多 5 次，指数退避，最大 3 秒延迟）。

## 九、常见问题

### Q: 页面显示 "请选择或创建一个项目"

首次使用时需要先创建项目。点击「+ 创建项目」开始。

### Q: 执行中看不到实时日志

检查底栏的连接状态是否显示「已连接」。如果断开，尝试刷新页面。

### Q: 代码 Tab 无法编辑

确认文件已加载完成。部分系统文件（如 `autoc.db`）为只读。

### Q: 预览面板空白

预览仅在生成 Web 应用时可用。确认项目包含 Web 服务且执行已完成。

### Q: Docker 沙箱启动失败

确保 Docker Desktop 正在运行。在设置抽屉的「沙箱管理」中检查容器状态。
