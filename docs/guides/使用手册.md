## 一、系统简介

AutoC 是一个**全自动软件开发系统**，模拟真实的软件开发团队，由三个 AI Agent 协作完成从需求分析到代码交付的全流程。

你只需要用自然语言描述你的需求，系统会自动完成：

```
你的一句话需求
     │
     ▼
┌────────────────┐  任务计划   ┌────────────────┐
│ PlanningAgent  │ ────────▶ │ CodeActAgent   │
│ 需求分析+规划   │            │ ReAct 循环：    │
│                │ ◀─ 反馈 ─ │ 编码→验证→修复  │
└────────────────┘            └───────┬────────┘
                                      │
                             验证通过 / 未通过
                                      │
                         ┌────────────┴────────────┐
                         ▼                         ▼
                    迭代修复循环              项目完成！
```

### 核心特点

| 特性 | 说明 |
|------|------|
| Agent 协作 | PlanningAgent 规划 + CodeActAgent 编码验证修复一体化，运作模式与 OpenHands 一致 |
| 全自动迭代 | 发现 Bug 自动修复，循环直到测试通过 |
| 多模型支持 | 支持 GLM-5、Kimi K2.5、GPT-4o、DeepSeek、Qwen 等 |
| Docker 沙箱 | 所有代码强制在 Docker 容器内执行，安全隔离 |
| Function Calling | Agent 通过工具读写文件、执行命令，真正动手 |
| Web 界面 | 可视化操作、实时监控、代码浏览、终端 |
| 需求智能优化 | 提交前自动评估需求质量，补全技术细节 |

通过 Web 界面使用：访问 http://localhost:3000，详细操作见 [Web界面指南.md](Web界面指南.md)。

## 二、环境准备

### 系统要求

| 依赖 | 要求 | 说明 |
|------|------|------|
| **操作系统** | macOS / Linux / Windows (WSL) | |
| **Python** | 3.12 及以上 | 核心运行时 |
| **Node.js** | 18 及以上 | MCP 服务器 + Web 前端 |
| **Docker** | Docker Desktop 或 Docker Engine | **强制依赖** — 所有项目在沙箱内执行 |
| **网络** | 能访问所选 LLM 的 API 接口 | |

### 环境检查

```bash
python3 --version    # 需要 3.12+
node --version       # 需要 v18+
docker info          # 确认 Docker 正在运行
```

> **Docker 是必须的。** AutoC 强制所有项目在 Docker 容器内执行以保证安全隔离，未安装 Docker 将无法运行任何开发任务。

### 获取 API Key

你至少需要一个 LLM 服务的 API Key：

| 服务商 | 获取地址 | Key 格式 |
|--------|----------|----------|
| **智谱 GLM** (推荐) | https://open.bigmodel.cn → API Keys | `xxx.yyy` |
| **Kimi Code** | https://www.kimi.com/code → 控制台 → API Keys | `sk-kimi-xxx...` |
| OpenAI | https://platform.openai.com/api-keys | `sk-xxx...` |
| DeepSeek | https://platform.deepseek.com/api_keys | `sk-xxx...` |
| 阿里通义千问 | https://dashscope.console.aliyun.com → API-KEY | `sk-xxx...` |

## 三、安装部署

### 一键安装

```bash
# 1. 进入项目目录
cd autoc

# 2. 创建虚拟环境并安装依赖
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# 3. 安装前端依赖（Web 界面需要）
cd web && npm install && cd ..

# 4. 安装 PM2（进程管理）
npm install -g pm2
```

### 验证安装

```bash
source .venv/bin/activate
docker info                # Docker 正常
node --version             # Node.js 正常
```

### 启动服务

```bash
# 开发环境（推荐）— 同时启动后端 API + 前端 UI
./scripts/pm2-manage.sh dev

# 启动后访问:
#   后端 API : http://localhost:8080
#   前端 UI  : http://localhost:3000
```

> 详细的 PM2 管理说明见 [PM2部署指南.md](PM2部署指南.md)。

## 四、配置详解

### 4.1 配置 API Key（必须）

打开 Web 界面 → 点击右上角设置图标 → 在「模型分配」区域选择 Provider → 填入 API Key → 保存。

配置持久化到 `config/models.json`，无需手动编辑文件。

### 4.2 中国区网络加速（可选）

设置环境变量 `AUTOC_USE_CN_MIRROR=1`（通过 PM2 配置或系统环境变量），Agent 安装依赖时自动使用国内镜像源：

启用后的效果：
- **Docker 沙箱容器**自动写入 `pip.conf`（清华源）、`.npmrc`（npmmirror）、`GOPROXY`（goproxy.cn）
- **Agent 提示词**注入镜像使用指南，pip/npm/go 命令自动携带镜像参数
- **环境修复**（缺失依赖自动安装）也会走镜像

不设置此变量时，所有包安装使用默认源。

### 4.3 选择 LLM 模型

AutoC 内置了 6 个预设，通过 Web 设置面板选择：

| 预设名 | 模型 | API 地址 | 对应环境变量 |
|--------|------|----------|-------------|
| `glm` | glm-5 | open.bigmodel.cn/api/paas/v4 | GLM_API_KEY |
| `kimi` | kimi-for-coding | api.kimi.com/coding/v1 | KIMI_API_KEY |
| `openai` | gpt-4o | api.openai.com/v1 | OPENAI_API_KEY |
| `qwen` | qwen3.5-plus | dashscope.aliyuncs.com/compatible-mode/v1 | DASHSCOPE_API_KEY |
| `deepseek` | deepseek-chat | api.deepseek.com/v1 | DEEPSEEK_API_KEY |
| `local` | default | localhost:8000/v1 | — |

### 4.4 混合模型配置

可以为每个 Agent 独立指定模型（推荐做法，详见 [模型选型指南.md](模型选型指南.md)）。

在 Web 设置面板的「模型分配」区域，为 PlanningAgent / CodeActAgent 分别选择 Provider 和模型。

## 六、运行流程详解

### Phase 0: 需求智能优化（可选）

**执行者**: RequirementRefiner

系统自动评估需求质量：
- **高质量** (≥70%)：跳过优化，直接进入 PM 阶段
- **中等质量**：静默增强 — AI 自动补全技术细节、明确范围
- **低质量** (<40%)：Web 界面显示质量评分和建议

可通过 `config.yaml` 的 `refiner.mode` 设置：`auto`（默认）/ `enhance`（强制增强）/ `off`（关闭）。

### Phase 1: 需求分析与项目规划

**执行者**: PlanningAgent

1. 深度理解需求，补充隐含功能点
2. 选择合适的技术栈
3. 设计系统架构和目录结构
4. 将项目拆解为细粒度的开发任务
5. 评估技术风险

**产出**: JSON 格式项目计划，包含任务列表、技术栈、架构说明。

### Phase 2: 编码+验证+修复

**执行者**: CodeActAgent

在 ReAct 循环中完成每个任务（支持并行执行）：
1. 按计划编写代码，创建文件
2. 在 Docker 沙箱内运行测试验证
3. 发现问题自动修复，循环直到验证通过
4. 通过 `submit_test_report` 工具提交结构化报告

## 七、Agent 角色说明

### PlanningAgent（规划）

| 属性 | 值 |
|------|-----|
| 可用工具 | 读取文件、列出目录、搜索文件 |
| 核心能力 | 需求分析、架构设计、任务拆解、风险评估 |

PlanningAgent 不走 ReAct 循环，单次 LLM 调用 + 纯代码验证。输出结构化 JSON 计划交给 CodeActAgent 执行。

### CodeActAgent（编码+验证+修复）

| 属性 | 值 |
|------|-----|
| 可用工具 | 读写文件、创建目录、搜索文件、执行命令、提交报告 |
| 默认最大迭代 | 由编排器动态调整 |
| 核心能力 | 代码生成、测试验证、Bug 修复，在 ReAct 循环中一体化完成 |

### CritiqueAgent（可选评审，默认关闭）

| 属性 | 值 |
|------|-----|
| 可用工具 | 读取文件、搜索文件、执行命令、提交评审 |
| 核心能力 | 4 维代码质量评分（correctness/quality/completeness/best_practices） |

需在 Web UI 设置中手动启用 `enable_critique`。

## 八、工具系统

Agent 通过 LLM 的 Function Calling 能力调用工具：

### 内置工具

| 工具 | 说明 | 权限 |
|------|------|------|
| `read_file` | 读取文件内容 | PM / Dev / Tester |
| `write_file` | 创建/覆盖写入文件 | Dev / Tester |
| `create_directory` | 创建目录 | Dev / Tester |
| `list_files` | 列出目录内容 | PM / Dev / Tester |
| `search_in_files` | 搜索文件内容 | PM / Dev / Tester |
| `execute_command` | 在 Docker 沙箱内执行命令 | Dev / Tester |

> **安全机制**: Shell 工具内置危险命令黑名单，自动拦截 `rm -rf /` 等操作。所有文件操作限制在项目目录内。

### MCP 扩展工具

通过 MCP 协议可集成更多工具（文件系统、Git、浏览器等），详见 [MCP快速入门.md](MCP快速入门.md)。

## 九、配置文件全字段说明

`config.yaml` 完整字段：

```yaml
# ==================== LLM 全局默认配置 ====================
llm:
  preset: "glm"             # 预设: kimi / glm / openai / qwen / deepseek / local
  base_url: ""              # API 地址（留空使用预设默认值）
  model: ""                 # 模型名（留空使用预设默认值）
  temperature: 0.7          # 生成温度 (0.0-2.0)
  max_tokens: 32768         # 单次最大生成 Token 数
  timeout: 120              # 请求超时（秒）
  max_retries: 3            # 最大重试次数
  retry_base_delay: 2.0     # 重试基础延迟（秒，指数退避）
  extra_params: {}           # 传递给 API 的额外参数

# ==================== Agent 独立配置 ====================
agents:
  planner:
    name: "PlanningAgent"
    description: "需求分析、任务拆解、架构设计"
    preset: glm               # 可独立指定预设
    model: glm-5              # 可独立指定模型
    timeout: 300              # 规划超时
    batch_size: 5             # 批量任务数
  code_act:
    name: "CodeActAgent"
    max_iterations: 10
    preset: glm
    model: glm-4.7

# ==================== 需求智能优化 ====================
refiner:
  mode: "auto"                 # auto=自动策略 | enhance=强制增强 | off=关闭
  quality_threshold_high: 0.7   # 高于此分数跳过优化
  quality_threshold_low: 0.4    # 低于此分数触发澄清

# ==================== 编排器配置 ====================
orchestrator:
  max_rounds: 3               # PM→Dev→Test→Fix 最多跑几轮
  max_iterations: 20          # 全局最大迭代次数
  auto_fix: true              # 测试失败后是否自动修复
  enable_checkpoint: false    # 启用断点保存
  context_limit: 25000        # 上下文窗口限制（Token）
  exit_detection:
    completion_threshold: 2
    require_exit_signal: true
    max_consecutive_done: 2
  circuit_breaker:
    no_progress_threshold: 3
    same_error_threshold: 5
    cooldown_minutes: 5
    auto_reset: false
  rate_limit:
    max_calls_per_hour: 100

# ==================== MCP 工具协议 ====================
mcp:
  enabled: true
  servers:
    - name: filesystem
      command: npx
      args: ["-y", "@modelcontextprotocol/server-filesystem", "{workspace}"]
    - name: browser
      command: npx
      args: ["-y", "@playwright/mcp@latest", "--output-dir", "/tmp/autoc-screenshots"]
  strategy:
    builtin_first: true
    show_in_prompt: true

# ==================== 功能开关 ====================
features:
  git: true
  sandbox_mode: "project"     # project=项目独占容器 | global=全局共享
  sandbox_image: "nikolaik/python-nodejs:python3.12-nodejs22"
  experience: true
  code_quality: true
  parallel: true
  max_parallel_tasks: 3
  progress_tracking: true
  venv:
    global_path: "~/.autoc/venv"
    project_venv: false

# ==================== 工作区配置 ====================
workspace:
  output_dir: "./workspace"
  clean_before_start: false
  isolation_mode: "multi"
  default_project_name: "project-{date}-{random}"

# ==================== 预览配置 ====================
preview:
  enabled: true
  auto_preview: "auto"
  runtime: "auto"

# ==================== 日志配置 ====================
logging:
  level: "INFO"
  file: "./logs/autoc.log"
  console: true
```

## 十一、进阶用法

### 11.1 备用 Provider 配置

在 `autoc-models.json` 的 `fallback` 段配置自动切换列表：

```json
{
  "fallback": {
    "pm": [
      { "provider": "qwen", "model": "qwen3.5-plus" },
      { "provider": "glm",  "model": "glm-4.7" }
    ],
    "developer": [
      { "provider": "kimi", "model": "kimi-for-coding" }
    ]
  }
}
```

主 Provider 所有重试耗尽后，自动按顺序尝试备用（401 认证错误除外）。

## 十二、常见问题 FAQ

### Q1: 报错 "未设置 API Key"

打开 Web 界面右上角设置，配置对应 Provider 的 API Key。

### Q2: 报错 "Docker 沙箱初始化失败"

确保 Docker Desktop 已启动。运行 `docker info` 检查状态。首次运行会拉取 `nikolaik/python-nodejs:python3.12-nodejs22` 镜像，需要网络。

### Q3: 报错 "LLM 调用失败: 401"

API Key 无效或过期，在设置面板重新配置并确保复制完整。

### Q4: 报错 "LLM 调用失败: 429"

API 调用频率超限。等待几分钟后重试，或切换到另一个模型预设。

### Q5: MCP 服务器连接失败

检查 Node.js 是否正确安装（`node --version` 需要 v18+）。MCP 服务器通过 `npx` 运行，首次会自动下载。

### Q6: 生成的代码有问题

1. 在 Web 界面点击「重新运行」
2. 优化需求描述
3. 切换更强的模型
4. 手动修改后点击「继续执行」重新验证

### Q7: 如何中断执行？

点击 Web 界面顶栏「停止」按钮。

### Q8: Token 消耗参考

| 项目规模 | 文件数 | 大约 Token |
|---------|--------|-----------|
| 简单 | 3-5 | 5,000-15,000 |
| 中等 | 10-15 | 20,000-50,000 |
| 复杂 | 20+ | 50,000-150,000 |

### Q9: 支持 Windows 吗？

支持通过 WSL 运行。

### Q10: 能同时用两个模型吗？

完全支持，在 Web 设置面板为每个 Agent 独立配置，详见 [模型选型指南.md](模型选型指南.md)。

## 十三、项目结构

```
autoc/                          # 项目根目录
├── config.yaml                 # 运行配置
├── requirements.txt            # Python 依赖
├── AGENTS.md                   # Cursor 工作区规则
│
├── autoc/                      # Python 包 — 全部源代码
│   ├── config.py               #   配置加载
│   ├── app.py                  #   Application Factory
│   ├── server/                 #   Web 服务 (FastAPI，含路由子模块)
│   ├── agents/                 #   Agent 实现 (PlanningAgent/CodeActAgent/CritiqueAgent)
│   ├── core/                   #   核心模块
│   │   ├── orchestrator/       #     编排器（调度/循环/门控/生命周期）
│   │   ├── project/            #     项目管理（状态/进度/模板/PRD导入）
│   │   ├── llm/                #     LLM 客户端（路由/缓存/上下文构建）
│   │   ├── analysis/           #     分析系统（复杂度/退出检测/失败分析）
│   │   ├── infra/              #     基础设施（熔断器/数据库/日志）
│   │   └── runtime/            #     运行时（Docker沙箱/预览/venv/MCP）
│   ├── tools/                  #   工具集 (文件/Shell/Git/沙箱)
│   └── stacks/                 #   技术栈适配器 (11 种)
│
├── web/                        # Web 前端 (React + Ant Design + Vite)
├── scripts/                    # 运维脚本 (PM2)
├── docs/                       # 文档
└── workspace/                  # 生成的项目输出 (gitignored)
```
