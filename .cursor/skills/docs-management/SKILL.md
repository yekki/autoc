# AutoC 文档管理

通过 4 个口令触发文档工作流，覆盖从零创建到持续维护的完整生命周期。

## 口令速查

| 口令 | 适用场景 | 跳转 |
|------|---------|------|
| **"初始化文档"** | 新项目 / 从零开始 | §工作流零 |
| **"更新文档"** | 代码改完，同步文档 | §工作流一 |
| **"产品对焦"** | 定期全量对焦 | §工作流二 |
| **"保存调研"** | 归档调研分析 | §工作流三 |

## 项目级资源

| 文件 | 作用 |
|------|------|
| `.cursor/docs-config.yaml` | 项目文档配置（目录结构、模板选项） |
| `docs/mapping.md` | 代码模块 → 产品方案的唯一权威映射表 |
| `docs/README.md` | 文档注册表（所有文档的状态索引） |

## §工作流零：初始化文档

**触发口令**：`"初始化文档"`

**前置条件**：项目有代码但无文档体系，或需要从零重建。

### 步骤

1. **扫描代码库结构**
   - 读取项目根目录，识别主要模块（`autoc/core/`、`autoc/agents/`、`autoc/tools/`、`web/` 等）
   - 读取 `requirements.txt` / `package.json` 识别技术栈
   - 读取现有 README / AGENTS.md 提取已有信息

2. **创建目录骨架**
   ```
   docs/
   ├── README.md            # 文档注册表
   ├── mapping.md           # 代码→文档映射
   ├── design/              # 产品方案
   ├── research/            # 调研分析
   ├── guides/              # 使用指南
   ├── benchmarks/          # Benchmark 复盘
   └── archive/             # 废弃归档
   ```

3. **生成 AGENTS.md**
   - 包含 Setup / Start / Test / Code Style / Security / Docs & Rules 六大节
   - 文档表格引用 `docs/` 下的实际文件

4. **为每个主要模块生成 design 文档**
   - 使用 [templates.md](templates.md) 中的 design 模板
   - 扫描对应代码提取：类/函数签名、数据模型、API 端点
   - 每篇包含：实现追踪表 + 变更记录表

5. **生成核心 guides**
   - 至少生成：使用手册 + Web界面指南
   - 交叉引用 design 文档

6. **生成 mapping.md**
   - 遍历所有代码模块，建立 代码路径 → design 文档 的映射
   - 建立 调研 → 产品方案 的映射

7. **生成 docs/README.md**
   - 注册所有新建文档，标注版本和状态

8. **校验**
   - 运行 `python .cursor/skills/docs-management/scripts/doc-validate.py`
   - 确认：每个 design 文档有实现追踪表，mapping.md 覆盖所有代码模块

## §工作流一：更新文档

**触发口令**：`"更新文档"`

**适用场景**：当前会话修改了代码，需同步更新文档。

### 步骤

1. **识别本次变更范围**
   - `git diff --name-only` 获取本次修改的文件列表
   - 如果无 diff（未 commit），向用户确认范围

2. **查映射表定位文档**
   - 读取 `docs/mapping.md` §二（快速查找表）
   - 确定需要更新的 design 文档和 guides

3. **更新 design 文档**
   - 读取对应 design 文档
   - 对比代码变更，更新涉及的章节
   - 更新实现追踪表中的状态
   - 追加变更记录

4. **联动 guides**
   - 根据 `.cursor/rules/docs.mdc` 中的 design→guides 联动映射表
   - 检查对应 guide 是否需要同步更新
   - 若需要则更新对应章节

5. **回写 research**（如适用）
   - 检查 `docs/mapping.md` §三（调研→产品方案映射）
   - 若改动实现了某调研建议，更新该调研的实现追踪表

6. **更新注册表**
   - 更新 `docs/README.md` 中对应文档的版本号
   - 更新 `AGENTS.md` 中的文档计数（如有新增/删除）

7. **更新 mapping.md**（如有新增模块/文档）

## §工作流二：产品对焦

**触发口令**：`"产品对焦"`

**适用场景**：定期全量检查代码与文档的一致性，补齐缺失。

### 步骤

1. **全量扫描代码模块**
   - 遍历 `autoc/` 和 `web/src/` 下所有 Python/JS 文件
   - 提取模块列表和关键类/函数

2. **对照 mapping.md 检查覆盖率**
   - 找出无对应 design 文档的代码模块 → 新建
   - 找出 design 文档中描述的但代码已删除的内容 → 标注过时

3. **逐篇审计 design 文档**
   - 对每篇 design 文档：
     a. 读取文档中的模块描述
     b. 读取实际代码
     c. 对比差异，列出过时/缺失内容
     d. 更新文档

4. **审计 guides**
   - 检查每篇 guide 引用的功能/配置是否仍然存在
   - 标注过时内容

5. **运行校验脚本**
   - `python .cursor/skills/docs-management/scripts/doc-audit.py`
   - 输出审计报告：覆盖率、过时项、缺失项

6. **生成对焦报告**
   - 汇总所有变更到 `docs/README.md` 顶部的变更记录

## §工作流三：保存调研

**触发口令**：`"保存调研"`

**适用场景**：当前会话产生了有价值的调研分析，需要归档。

### 步骤

1. **确认调研主题和范围**
   - 向用户确认调研标题
   - 确认关联的产品方案文档

2. **使用 research 模板创建文档**
   - 使用 [templates.md](templates.md) 中的 research 模板
   - 文件命名：中文描述 `.md`（如 `Token优化研究.md`）
   - 放到 `docs/research/` 目录

3. **填充内容**
   - 必须包含：调研背景、调研范围、分析过程、结论
   - 必须包含：**改进建议**（含优先级 P0-P3）
   - 必须包含：**实现追踪表**（每条建议的实现状态）
   - 必须包含：变更记录

4. **标注关联**
   - 在 `docs/mapping.md` §三 添加映射记录
   - 在关联的 design 文档的实现追踪表中添加"来源"引用

5. **更新注册表**
   - 在 `docs/README.md` 的调研分析表中添加条目

## 格式规范速查

| 规则 | 说明 |
|------|------|
| 语言 | 中文简体 |
| 标题层级 | `#` 仅文档标题，正文从 `##` 开始 |
| 代码块 | 必须语法高亮（```python / ```bash / ```json） |
| 对比 | 优先用表格 |
| 引用 | 代码/命令/文件名用 `反引号` |
| design 必含 | 实现追踪表（含 `来源` 列）+ 变更记录表 |
| research 必含 | 改进建议（含优先级）+ 实现追踪表 + 变更记录 |
